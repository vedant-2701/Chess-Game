# Step 2
Three decisions in these migrations worth being explicit about:
- users.id is not DEFAULT gen_random_uuid() — the client generates the UUID and submits it. The server inserts it as-is (the CreateOrGetUser upsert in Step 3). If the server generated it, you'd need an extra round-trip to tell the client its own ID. No default is the correct choice here.
- player_white_id is NOT NULL, player_black_id is nullable — White is set at game creation (POST /games). Black is set when someone joins (POST /games/:id/join). A game row can exist without Black for its entire WAITING_FOR_PLAYER lifetime. Making black nullable at the DB layer accurately reflects this lifecycle.
- The idx_moves_game_id index alongside the unique index — technically PostgreSQL can use the leading column of idx_moves_game_move for game_id-only lookups via a partial index scan. The explicit idx_moves_game_id makes the intent unambiguous and avoids a correctness dependency on planner behavior. In Phase 4 when you run EXPLAIN ANALYZE on GetMovesForGame, you'll see which index is actually used and can drop the redundant one if warranted. For now, explicit is better.

# Step 3
- scanGame takes func(dest ...any) error — both pgx.Row.Scan and pgx.Rows.Scan satisfy this signature, so one helper handles single-row and multi-row queries without duplicating column scan order. This matters because scan order must exactly match the SELECT column list; having two copies is a maintenance hazard.
- Nullable columns scanned as *string then converted — pgx/v5 can scan NULL into *string reliably. Scanning directly into *GameStatus or *Outcome (user-defined string types) involves pgx's reflection path, which is less predictable across versions. Scanning as *string and converting explicitly is two extra lines per nullable field but zero ambiguity.
- GetMovesForGame returns non-nil empty slice — make([]*Move, 0) not var moves []*Move. The game layer that calls this will range over the result and send it in a GAME_STATE message. A nil slice serializes to null in JSON; an empty slice serializes to []. The client needs [] for a game with no moves yet.
- UpdateGameStatus takes *GameOutcome not *Outcome — the spec suggested *Outcome but you cannot write an outcome to the DB without also writing an outcome_reason; the schema's CHECK constraint enforces that both are present or both are null. Splitting them across two parameters would make it impossible to maintain the invariant at the call site. *GameOutcome enforces the invariant in the type system.

# Step 4
Three things worth understanding in this implementation:
- The keyFunc algorithm check is not optional. Without if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok { return nil, err }, a crafted token with "alg": "none" would bypass signature verification entirely — the library would accept it as valid. This is a known class of JWT vulnerability. The check costs one line and eliminates the entire attack surface.
- ErrTokenExpired is separate from ErrTokenInvalid because the WebSocket handler needs to distinguish them. An expired token means the player had a valid session that timed out — the right response is a specific error code (INVALID_TOKEN with message "token expired") so the client knows to request a new one. An invalid token means something is wrong at a more fundamental level. Collapsing both into one error loses that distinction at the call site.
- fmt.Errorf("%w: %v", ErrTokenInvalid, err) — ErrTokenInvalid is the sentinel wrapped with %w so errors.Is works. The underlying jwt error is appended with %v so it appears in logs for debugging, but is not itself wrapped (it doesn't need to be unwrappable by callers). This is the correct pattern when you want a stable public sentinel plus diagnostic detail.
